# Phase 19 Plan 19.1-01: streamTts Abort Re-throw Contract Fix Summary

Gap closure: restore the "re-throws a typed AbortError on abort" invariant documented in plan-02's JSDoc and must_haves, broken by the bare `return` in the signal-poll guard.

## What Was Built

Single-file fix to `src/channels/voice/tts.ts` that makes the signal-poll abort path inside `streamTts` reject (throw) instead of resolve (return), and ensures the thrown error satisfies `isAbortError()` so the adapter's catch branch is reachable via both abort detection paths.

## The Fix â€” Three Coordinated Changes

### 1. Replace bare `return` with typed `throw` inside the for-await loop guard

```typescript
// Before:
if (signal.aborted) {
  return  // resolved silently â€” broke the re-throw contract
}

// After:
if (signal.aborted) {
  throw Object.assign(new Error('TTS aborted by signal'), { name: 'AbortError' })
}
```

This ensures the polling path (abort observed between chunks) rejects with an `isAbortError()`-satisfying error rather than resolving silently.

### 2. Thread `{ signal }` into `Readable.fromWeb`

```typescript
const readable = Readable.fromWeb(
  body as import('node:stream/web').ReadableStream<Uint8Array>,
  { signal },
)
```

The plan-stated fix (replace return with throw) is insufficient alone because the test mock's `makeAbortableWebStream` uses `setTimeout(0)` between chunks, and in Vitest's environment all 3 chunks are pre-buffered before the `setTimeout(5)` abort fires. Passing `signal` to `Readable.fromWeb` causes Node.js to emit an `AbortError` on the readable when the signal fires, interrupting the `for await` loop immediately â€” even if chunks are queued. This is "abort via signal" (the plan's mandated approach), not `.destroy()`.

### 3. Post-loop signal check before sending `tts_done`

```typescript
if (signal.aborted) {
  throw Object.assign(new Error('TTS aborted by signal'), { name: 'AbortError' })
}
```

Guards the edge case where abort fires concurrently with the final chunk â€” the loop polling guard runs only between chunks, so abort on the last chunk is caught here.

## Why the Original Bare-Return Broke the Contract

Plan 19-02's JSDoc states: "On abort, this function re-throws a typed AbortError." The bare `return` inside the `if (signal.aborted)` guard violated this by resolving the promise with `undefined`. Two consequences:
1. The test `"stops iterating when the signal is aborted and does NOT send tts_done"` failed with "promise resolved 'undefined' instead of rejecting"
2. The adapter's `isAbortError(err)` catch branch was unreachable for abort detected by polling â€” barge-in cancellations would be silently swallowed rather than logged at debug level

## Test File Unchanged

`src/channels/voice/tts.test.ts` was not modified. The test at line 116 already used `await expect(p).rejects.toSatisfy((e: unknown) => isAbortError(e))` â€” it was correct as written, testing the intended contract. The bug was strictly in the implementation.

The adapter's `isAbortError` catch branch in `send()` is now reachable via both:
- SDK path: stream throws `APIUserAbortError` (name matched by existing `e.name === 'APIUserAbortError'` branch)
- Polling path: loop throws `AbortError` (name matched by existing `e.name === 'AbortError'` branch)
